IDLE MOMENTS BLOG APP - BEHAVIOR-ONLY SPEC

Purpose
This document describes the complete page-by-page and component-by-component behavior of the current Idle Moments blog/forum application. It intentionally excludes visual styling instructions so another model or designer can recreate the UI from scratch with a new visual direction.

Application Summary
Idle Moments is a hobby-focused blog and discussion app. Users can browse a feed of posts, browse topic threads, read individual post details, reply to comments, star posts or threads, create/edit posts, edit profile information, register/login/recover an account, and view profile/admin pages. The current app is a front-end prototype with mock auth state, fallback data, and local-only form behaviors in most places.

Global Routing
- The app is a React single-page app using BrowserRouter.
- The router basename is derived from Vite's BASE_URL. In the current deployment, BASE_URL is "/frontend/", so all browser routes should work under "/frontend".
- The root path "/" redirects to "/links".
- Every routed page renders inside a global page shell that includes the site navigation above the page content.

Defined Routes
- "/" redirects to "/links".
- "/links" displays a page index linking to every route.
- "/home" displays the public landing page.
- "/feed" displays the post feed.
- "/threads" displays the list of all threads.
- "/view-thread" displays a single thread with its posts.
- "/post-details" displays a public/non-owner post detail page.
- "/my-post-details" displays the current user's owned post detail page.
- "/my-profile" displays the current user's profile and posts.
- "/user-profile" displays another user's profile and posts.
- "/admin-profile" displays the current user's profile/posts plus admin list editing controls.
- "/create-post" displays the create post form.
- "/edit-post" displays the edit post form.
- "/edit-profile" displays the edit profile form.
- "/login" displays the login form.
- "/register" displays the registration form.
- "/forgot-password" displays the account recovery form.
- "/new-password" displays the change password form.

Mock Auth State
- The app has a mock logged-in user:
  - username: "AndMarsh"
  - avatarVariant: "blue"
- All routes are treated as logged in except "/feed".
- On "/feed" only, the nav renders in logged-out mode for testing.
- No actual auth API is wired in the current front end.

Small-Screen Drawer Behavior
- At the small-screen breakpoint, desktop links are replaced by a hamburger button.
- Hamburger button:
  - aria-label is "Open navigation menu" when closed.
  - aria-label is "Close navigation menu" when open.
  - aria-expanded reflects open state.
  - aria-controls points to the drawer.
- Clicking the hamburger opens/closes the drawer.
- Drawer contents when logged in:
  - Profile avatar link at the top, navigating to "/my-profile".
  - Feed -> "/feed".
  - Threads -> "/threads".
  - Write -> "/create-post".
- Drawer contents when logged out:
  - No profile avatar.
  - Feed -> "/feed".
  - Threads -> "/threads".
  - Login -> "/login".
- Drawer has a close button with aria-label "Close navigation menu".
- Clicking any drawer link closes the drawer.
- Clicking the backdrop closes the drawer.
- Pressing Escape closes the drawer.
- When drawer is open, document body scrolling is disabled.
- Drawer unmounts after a short closing delay so close animation can finish.
- On route change, the drawer closes automatically.

Shared Data Model: Post Preview
Post preview objects should support:
- id: string.
- title: string.
- authorUsername: string.
- stars: number.
- comments: number.
- excerpt: string.
- publishedAt: string or parseable date.

Fallback Post Preview Data
- There are three fallback posts:
  - "How to Live like a Robot" by "robotfan42", 18 stars, 6 comments.
  - "How to Float in a Pool" by "poolnoodle", 9 stars, 3 comments.
  - "Introspection for Adults with Autism" by "thoughtful_dev", 27 stars, 11 comments.
- All fallback posts have excerpts and timestamp strings.

Post Normalization Behavior
- A posts API response may be:
  - an array,
  - an object with posts,
  - an object with data,
  - an object with items.
- Each post may use different field names:
  - id or postId.
  - title or name.
  - authorUsername, username, author.username, or author.name.
  - excerpt, summary, body, content, or description.
  - publishedAt, createdAt, timestamp, or date.
  - stars, starCount, or likes.
  - comments, commentCount, replies, or replyCount.
- Invalid posts without a title are ignored.
- If no valid posts exist, use fallback posts.
- Missing author should display as "unknown".
- Missing excerpt or publishedAt should display as empty text.
- Number fields should be coerced to numbers.

Timestamp Formatting
- If timestamp is empty, display empty string.
- If timestamp cannot be parsed as a Date, display the original string.
- If parseable, format in US short numeric date/time:
  - month numeric
  - day numeric
  - two-digit year
  - hour numeric
  - minute two-digit

Shared Hook: usePosts
Inputs
- endpoint: string or falsy.
- options.fallback: optional fallback post array.

Behavior
- If endpoint is falsy:
  - posts becomes fallback.
  - loading becomes false.
  - error becomes null.
- If endpoint exists:
  - loading starts true.
  - error starts null.
  - fetch endpoint.
  - If response is not ok, throw an error.
  - Parse JSON.
  - Normalize posts and set posts.
  - On failure, set posts to fallback and set error.
  - Always set loading false when finished.
- Ignore async results after component unmount.
- The current UI generally does not render loading/error states, but the hook exposes them.

Shared Component: PostForm
Used by "/create-post" and "/edit-post".

Inputs
- title: form heading.
- submitLabel: submit button text.
- initialValues: optional object with title, thread, body.
- onSubmit: optional callback receiving the form state.

Form State
- title: string.
- thread: string.
- body: string.

Default Empty State
- title: "".
- thread: "".
- body: "".

Thread Loading Behavior
- On mount, fetch "http://localhost:8080/api/threads".
- If successful, normalize the result into thread options.
- If failed, use fallback thread options:
  - General
  - Questions
  - Show and Tell
- Loading state controls select placeholder:
  - "Loading threads..." while loading.
  - "Select a thread" after loading.
- Ignore async result after unmount.

Thread Option Normalization
- API payload may be:
  - array,
  - object with threads,
  - object with data,
  - object with items.
- Thread may be a string or object.
- Object id may come from id, threadId, value, or index.
- Object name may come from name, label, title, or threadName.
- Invalid options without names are discarded.
- If none valid, use fallback options.

Fields
- "Post Title" text input:
  - id/name: postTitle/title.
  - required.
  - controlled by form.title.
- "Thread" select:
  - controlled by form.thread.
  - required status is not explicit in current code, but should be treated as needed for a valid post.
- "Post Body" textarea:
  - id/name: postBody/body.
  - rows: 8.
  - required.
  - controlled by form.body.

Submit Behavior
- Prevent default.
- Calls onSubmit(form) if onSubmit exists.
- Current create/edit pages do not pass onSubmit, so no persistence occurs from those pages.

Shared Component: AdminEditableList
Used only in "/admin-profile".

Inputs
- title: list heading.
- endpoint: API endpoint for the list.
- fallbackItems: array of {id, name}.
- newItemLabel: label for add-item input.

State
- items: current list.
- newName: text input for new item.
- editingId: id of item currently being edited.
- editingName: temporary value for item edit.

Load Behavior
- On mount and when endpoint/fallback/title changes:
  - fetch(endpoint).
  - If successful, normalize items.
  - If failed, use fallbackItems.
  - Ignore async result after unmount.

Item Normalization
- API payload may be:
  - array,
  - object with data,
  - object with items,
  - object with mascots,
  - object with threads.
- Item may be a string or object.
- Object id may come from id, mascotId, threadId, value, or index.
- Object name may come from name, label, title, mascotName, or threadName.
- Invalid items without a name are discarded.
- If none valid, use fallbackItems.

Display Behavior
- Renders an ordered list.
- Each item displays:
  - 1-based index followed by a period.
  - item name, unless currently editing.
  - edit button, unless currently editing.
- Edit button:
  - aria-label: "Edit {item.name}".
  - Clicking sets editingId and editingName.
- Editing mode:
  - Replaces item name with input controlled by editingName.
  - Shows save button.
  - Save button aria-label: "Save {item.name}".
- Save behavior:
  - Trim editingName.
  - If empty, do nothing.
  - PATCH `${endpoint}/${itemId}` with JSON body { name: nextName }.
  - Regardless of PATCH success/failure, update local item name, clear editingId, clear editingName.
- Add behavior:
  - Input is controlled by newName.
  - Add button aria-label: "Add {newItemLabel}".
  - Trim newName.
  - If empty, do nothing.
  - Create placeholder item with id `placeholder-${Date.now()}` and name nextName.
  - POST endpoint with JSON body { name: nextName }.
  - If POST response is ok and JSON can be parsed, normalize returned payload and use it as new item.
  - Regardless of POST success/failure, append next item locally and clear newName.

Comment Normalization
- Payload may be array or object with comments, data.comments, or data.
- If no valid comment array exists, use fallback comments.
- Parent relationships:
  - Build a map of comments by id.
  - Each comment's parent id comes from parentComment.
  - If parent exists, create replyingTo summary object from parent.
- Comment fields:
  - id from id or index.
  - username from username, authorUsername, author.username, or "unknown".
  - body from body or content.
  - publishedAt from publishedAt or createdAt.
  - color defaults to "blue".
  - replyingTo is null or summary object.

Important Current Prototype Limitations
- Most form submissions only call preventDefault and do not persist data.
- Create Post and Edit Post do not persist because no onSubmit is passed.
- Login, Register, Forgot Password, New Password, and Edit Profile do not call APIs on submit.
- Comment submission does not add comments.
- Delete Comment actions do not delete comments.
- Star toggles are local visual state only and do not update counts or call APIs.
- Nav auth state is mocked; only /feed is forced logged out.
- Many API endpoints point to localhost and are expected to fall back when unavailable.
